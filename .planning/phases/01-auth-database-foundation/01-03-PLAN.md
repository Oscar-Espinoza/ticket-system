<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> GitHub OAuth -> /api/auth/callback/github | OAuth code/state crosses back into the app |
| account table -> server render | Stored access token must never reach the client |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-09 | Spoofing / CSRF | OAuth state parameter | mitigate | Better Auth manages OAuth state + PKCE + callback (do not hand-roll); socialProviders.github only |
| T-01-10 | Information disclosure | GitHub token to client / session | mitigate | D-05 — session carries only derived githubConnected boolean; token read only via getGitHubToken server-side; dashboard never embeds the token |
| T-01-11 | Information disclosure | Token at rest (plaintext) | accept | D-03 — token has only read:user/user:email in Phase 1; getGitHubToken (D-04) is the seam for Phase 7 AES-256-GCM |
| T-01-12 | Elevation of privilege | Over-broad OAuth scopes at signup | mitigate | D-01/D-02 — scope locked to read:user/user:email; repo/admin:repo_hook deferred to Phase 7 |
</threat_model>
