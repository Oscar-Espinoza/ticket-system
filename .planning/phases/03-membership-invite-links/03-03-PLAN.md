<objective>
Vertical slice: a public `/invite/[token]` landing page that resolves the token to
a project and, for a logged-in user, offers an explicit "Join project" button that
runs the idempotent `joinProject` Server Action. The action redirects to the
project on success (navigation is owned by the Server Action via `redirect()`, per
03-PATTERNS.md). Logged-out visitors get a sign-in CTA that returns to the invite
link; invalid or expired tokens get a clean error page. Cuts UI → Server Action → DB.

Purpose: MEM-02 (accept invite, join as member) with idempotent re-visit (D-29)
and POST-only join (D-27 — never a GET side effect). This is the first PUBLIC
(un-guarded) app route (D-26/D-28).
Output: join.ts action, join-project-button.tsx, invite/[token]/page.tsx.
</objective>

Server Action conventions (src/app/actions/projects.ts):
- session via `auth.api.getSession({ headers: await headers() })`
- `crypto.randomUUID()` ids; `db.insert(...)` (single insert here, no batch needed)
- 23505 detection: `(err as {code?})?.code ?? (err as {cause?:{code?}})?.cause?.code`; re-throw other codes
- `redirect()` from 'next/navigation' for the success navigation (NOT router.push). NOTE:
  `redirect()` works by throwing a special NEXT_REDIRECT error — it must be called
  OUTSIDE the try/catch that maps DB errors, otherwise the catch swallows it.

Public-page session pattern (src/app/(auth)/login/page.tsx + projects/[id]/page.tsx):
- `auth.api.getSession({ headers: await headers() })` may return null — do NOT

Login-return: Phase 1 login page accepts `?redirect=` (match its existing param;

    - With a valid unexpired token + new user: inserts one project_member row role 'member', then redirect()s to /dashboard/projects/${projectId}

    `{ error: 'invalid' }` (do NOT leak project details, D-28). Otherwise derive

    check-then-insert — query project_member for (projectId, session.user.id); if a

    map SQLSTATE 23505 (concurrent double-join) to a no-op (treat as already a member
    — the race-safe backstop, D-29); re-throw any other code. (Re-throwing
    non-23505 errors is REQUIRED.)

    After the try/catch succeeds, on every success path (fresh join, already-member,
    and 23505-backstop), call `revalidatePath(\`/dashboard/projects/${projectId}\`)`

    be OUTSIDE the DB try/catch (it throws NEXT_REDIRECT, which the DB catch would
    otherwise swallow). The Server Action OWNS navigation — do NOT return a projectId
    for the client to navigate with, and do NOT delegate navigation to a client
    useEffect (mirrors createProject / 03-PATTERNS.md).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| public internet → /invite/[token] | first un-guarded route; token + session both untrusted |
| link prefetchers / unfurlers → invite URL | bots may GET the URL; must not auto-join |
| client → joinProject action | token from FormData untrusted; concurrent double-submit possible |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-08 | Tampering | auto-join on visit | mitigate | Join is an explicit POST Server Action triggered only by the submit button (D-27); GET on /invite/[token] only renders the landing — no mutation on a safe method. |
| T-03-09 | Information Disclosure | invalid token page | mitigate | State C returns 200 with a generic message and no project internals; never notFound() (avoids confirming token existence) per D-28. |
| T-03-10 | Tampering | duplicate membership under concurrent join | mitigate | check-then-insert + 23505 backstop on the live unique(project_id,user_id) constraint → second insert resolves to "already a member" → redirect (D-29). |
| T-03-11 | Elevation of Privilege | expired link reuse | mitigate | Invitation lookup filters `expiresAt > now`; expired tokens yield State C / { error: 'invalid' } and never insert (D-24). |
| T-03-SC | Tampering | package installs | accept | No package-manager installs in this plan. |
</threat_model>
