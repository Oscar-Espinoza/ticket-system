<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| authenticated client → `/dashboard/projects/[id]` Server Component | `id` is attacker-controllable; the user may not belong to that project (IDOR surface). |
| page → neon-http `db` | The project SELECT must never run before membership is confirmed. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-10 | Information Disclosure (IDOR) | `/dashboard/projects/[id]` | mitigate | `requireProjectMember(id, userId)` is the FIRST authorization call; on failure the page calls `notFound()` (404, not 403) so it does not confirm a project's existence to outsiders. Throw precedes the project SELECT — the 403-before-DB guarantee (MEM-06, success criterion #4). Verified by the PROJ-03 test. |
| T-02-11 | Elevation of Privilege | catch block swallowing notFound() / unexpected errors | mitigate | `if (err instanceof ProjectAccessError) notFound(); throw err` — re-throws non-domain errors and lets notFound()'s control-flow exception propagate (pitfall #3). |
| T-02-12 | Spoofing | session bypass via middleware (CVE-2025-29927) | mitigate | Authorization lives in the page Server Component, not `middleware.ts`. No `forbidden()`/`authInterrupts` experimental path used (D-15). |
</threat_model>
