<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client form → `createProject` Server Action | User-supplied name + ticketKey are untrusted; client transform is UX only. |
| Server Action → neon-http `db` (batch) | Two-row insert must be atomic to avoid an ownerless project. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-03 | Tampering (data integrity) | ownerless-project on partial batch failure | mitigate | `db.batch([insert project, insert owner member])` is atomic on neon-http — both roll back if either fails. No sequential awaits. Verified by the PROJ-01 atomic test. |
| T-02-04 | Tampering | ticket-key injection (non-uppercase / special chars) | mitigate | Server-side `/^[A-Z]{2,6}$/` test before any INSERT; client transform is presentation only. Verified by the PROJ-01 invalid-key test. |
| T-02-05 | Spoofing | session source for creator id | mitigate | Creator id comes from `auth.api.getSession({ headers: await headers() })` on the server, never from client input. |
| T-02-06 | Denial of Service / data leak | unexpected DB errors | mitigate | Only `23505` is mapped to a field error; all other errors are re-thrown (not swallowed), surfacing failures rather than corrupting state. |
</threat_model>
